# with-arivie-chat — implementation notes

Anchor: forked from vercel/chatbot @ Apache-2.0 (commit at clone time).
Goal: production-shaped Next.js chat for Arivie analytics agents.
Mastra-native throughout. NO Drizzle for chat state. Better Auth instead of Auth.js.

## Scope contract

- Next.js 15 App Router · AI SDK 6 · Mastra · Better Auth · shadcn/ui
- handleChatStream({ mastra, agentId }) is the wire-in point
- Mastra Memory threads + messages (memory.createThread / memory.recall)
- Direct providers (Gemini / OpenAI / xAI / Anthropic), NO AI Gateway
- Tests: live against Gemini 2.5 Flash + agent-browser for the browser flow

## Decisions / deviations

### Production deploy — Vercel + Neon
- Live at https://arivie-chat-demo.vercel.app — verified end-to-end:
  sign-up, session cookie, /api/chat streaming with Gemini 2.5 Flash,
  multi-turn memory recall ("PONG" → "what was the word?" → "PONG"),
  artifact pane (demo button) all working.
- **Vercel monorepo wiring** — the example is at `examples/with-arivie-chat/`
  in a pnpm workspace. The Vercel project's Root Directory is set to that
  path; `vercel.json` IN THE SUBDIR (not the repo root — the repo root's
  vercel.json belongs to the docs project) carries:
  ```
  framework: nextjs
  buildCommand: cd ../.. && pnpm install --frozen-lockfile && pnpm build
                 && pnpm --filter with-arivie-chat build
  installCommand: cd ../.. && pnpm install --frozen-lockfile
  outputDirectory: .next
  ```
  When linking via CLI, `vercel.json` settings override project settings;
  precedence is `subdir/vercel.json` > project settings > repo-root
  `vercel.json`. Root-level rootDirectory was set via the Vercel REST
  API (`PATCH /v9/projects/{id}`) since the CLI's `vercel link` defaults
  to the linked dir.
- **Workspace filesystem on Vercel** — lambda fs is read-only outside
  `/tmp`. `lib/arivie.ts` was creating `.arivie/workspace` under
  `process.cwd()` which is `/var/task/...` on Vercel → ENOENT on every
  request. Fix: when `process.env.VERCEL === "1"`, root the workspace at
  `/tmp/arivie/workspace`. Local dev unchanged.
- **Database** — Neon Postgres free tier, region us-east-1, pooled
  connection string for serverless. Schema bootstrap: `pnpm setup` for
  the read-only role + owner identity, then `npx @better-auth/cli migrate
  --y` for the auth tables. Both run once per database.
- **Environment variables** — `BETTER_AUTH_URL` and `NEXT_PUBLIC_APP_URL`
  must match the production alias (`https://arivie-chat-demo.vercel.app`),
  not the random per-deploy URL. Chicken-and-egg: first deploy generates
  the alias; set the env vars; redeploy. Better Auth rejects cookies if
  the URL doesn't match.
- **Postgres pooling** — Mastra Memory writes go through `postgres` (with
  the pooled URL), Better Auth uses `pg` Pool against the same URL.
  Single-database for chat state + auth + analytics is the design.

### Phase 2 — artifact subsystem
- Artifact shape is a discriminated union (`query | chart | report |
  entity`) with zod schemas. See `lib/artifacts.ts`.
- Artifacts arrive on the client via three paths, all unified by chat-shell:
  (1) `data-artifact-*` parts the server writes into the UI message stream
  (forward-compat — handleChatStream v6 lets a custom route inject these),
  (2) `tool-<name>` parts whose outputs match a known artifact shape
  (`detectArtifact` heuristic — auto-upgrades execute_<source> outputs to
  query artifacts and write_report outputs to report artifacts without
  forcing any server change), (3) local demo state (the sidebar's "Demo
  artifacts" button — proves the renderer pipeline works without any
  real tool firing).
- `components/artifact-pane.tsx` ships one renderer per kind:
  - QueryView — dialect badge, SQL block, row/duration, explanation.
  - ChartView — Vega-Lite spec as JSON; the host app wires vega-embed or
    @observablehq/plot. Spec is the contract; rendering is the host's call.
  - ReportView — markdown via Streamdown, workspace path in header.
  - EntityView — labeled field cards with format hints (currency, date,
    datetime, number, percent).
- Browser-tested all four renderers via the demo button at 1600×900
  viewport. Screenshots at `.screenshots/phase2-3-{query,chart,report,
  entity}.png`. Tabs work; the only UX wart is that the tab strip
  overflows when there are 4+ tabs and the close-button takes priority —
  acceptable for the starter, host apps can replace the layout.
- agent-browser quirk: clicks on horizontally-clipped buttons in
  overflow-x-auto containers don't always fire. Workaround for tests:
  use `agent-browser eval` to dispatch the click via JS. Documented for
  anyone wiring CI tests against this UI.

### Phase 1.6 browser smoke test — what mattered
- `agent-browser fill` writes the input value via direct DOM assignment,
  which doesn't fire React's synthetic onChange. The textbox visually has
  the text but `useState` is still "". The Send handler bails on the empty
  guard and no POST hits `/api/chat`. Fix: use `agent-browser type` (real
  keystrokes) — that goes through the React event pipeline. Documented for
  anyone scripting headless tests against this starter.
- Mastra v1's `memory.getThreadsByResourceId` returns
  `{ threads, total, page, perPage, hasMore }` — NOT a bare array (changed
  from earlier versions). ThreadList.tsx assumed array. Normalised in
  `app/api/threads/route.ts` to flatten before returning.
- Screenshot of working chat at `.screenshots/phase1-6-smoke.png`.

### Phase 1.5 live test — what actually mattered
- `handleChatStream` defaults `version: "v5"`, but our client is AI SDK 6
  (`@ai-sdk/react@^3` + `ai@^6`). Memory writes worked under v5, but the
  agent did NOT receive the lastMessages history on subsequent turns — the
  assistant just echoed the user's prompt. Fix: pass `version: "v6"`
  explicitly in `app/api/chat/route.ts`. After the switch, turn-2 recall
  works (turn 1: "my favorite color is teal" → turn 2: "what is my
  favorite color?" → "is teal"). This is the single non-obvious wire-up
  fact for anyone forking this starter.
- `new Memory()` in `@arivie/agent` is fine as-is. Default `lastMessages`
  is 10 (Mastra v1 default; was 40 pre-v1). Storage inherits from the
  Mastra instance's PostgresStore, so messages persist into the shared
  `arivie_chat` db alongside Better Auth tables.
- Better Auth schema migration: `npx -y @better-auth/cli@latest migrate
  --y` creates `user`, `session`, `account`, `verification` tables in the
  same db. Wired up as `pnpm exec` was wrong — the package isn't in
  this workspace; `npx` pulls it transiently. Adding a `pnpm setup` script
  for the role+owner identity bootstrap; the auth migrate is a one-line
  npx in the starter's README.
- TSX `TransformError` on top-level await scripts: esbuild defaults to CJS
  output for `.ts` files run inline via `tsx -e`. Fix: write the script as
  a real file with an `async function main() {} ; main().catch(...)`
  pattern. `scripts/setup-db.ts` follows this.
- Chat `/api/chat` route file was missing after the earlier fork pass
  (likely never landed in the surgical refactor). Recreated.
- Default thread title is empty string in the threads list response —
  fine for now; the agent's auto-title generation is disabled by default
  in Mastra v1. If we want titles, set `generateTitle: true` on the
  Memory instance.
