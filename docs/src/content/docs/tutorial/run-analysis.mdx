Northwind has two query tools. **`execute_postgres`** runs a read-only SELECT you write (or the model writes) against the warehouse. **`compile_metric`** compiles a declared semantic measure — revenue, order count, average order value — into canonical SQL and runs it. Both pass through the same SELECT-only guard and the same read-only role.

Use `execute_postgres` when you need ad-hoc exploration:

```text
Which customer has the highest total spend?
```

The model writes a grouped SELECT, calls `execute_postgres`, reads the rows, and answers.

Use `compile_metric` when the question maps to a declared measure:

```text
What was revenue over the last 30 days?
```

The agent calls `compile_metric` with `entity: "orders"`, `measure: "revenue"`, `segment: "last_30_days"`. The compiler expands the measure's canonical SQL from the semantic layer:

```sql
SUM(amount_cents) FILTER (WHERE status = 'paid') / 100.0
```

…and applies the segment filter:

```sql
created_at >= CURRENT_DATE - INTERVAL '30 days'
```

One guarded query runs; the tool returns a single number — for the seeded tutorial data, **$4,448.96**. The model narrates that result; it does not recompute it.

## Math discipline

LLMs are unreliable at multi-step arithmetic. If the model pulls revenue for this week, pulls revenue for last week, then computes the percentage in its reply, it will silently get it wrong on some runs.

The fix: **every arithmetic step lives in one SQL query**. Chain CTEs so a single `execute_postgres` or `compile_metric` call returns the final shape — value, delta, percentage, verdict — and the model only forwards the rows.

```sql
WITH this_week AS (
  SELECT SUM(amount_cents) FILTER (WHERE status = 'paid') / 100.0 AS revenue
  FROM orders
  WHERE created_at >= CURRENT_DATE - INTERVAL '7 days'
),
last_week AS (
  SELECT SUM(amount_cents) FILTER (WHERE status = 'paid') / 100.0 AS revenue
  FROM orders
  WHERE created_at >= CURRENT_DATE - INTERVAL '14 days'
    AND created_at <  CURRENT_DATE - INTERVAL '7 days'
)
SELECT t.revenue,
       ROUND(100.0 * (t.revenue - l.revenue) / NULLIF(l.revenue, 0), 1) AS revenue_wow_pct
FROM this_week t CROSS JOIN last_week l;
```

One tool call. Exact math in Postgres; the reply cites the rows.

Right now the model could reinvent "revenue" every turn — different filters, different date math. Pin the definition so `"revenue last week"` always compiles the same way in [Remember Definitions](./remember-definitions).
