`defineArivie()` builds **Mastra agents** from your `agents` map. Each agent gets the tools and instruction fragments from the plugins backing its `capabilities` — for analytics, that means text-to-SQL (`compile_metric`, `execute_<source>`) plus any workspace tools your deployment wires. The agent that runs the SQL is the same agent that writes the Markdown report. No supervisor, no sub-agents, no inter-agent prose handoff.

```ts
const arivie = await defineArivie({
  app: { id: "acme", name: "Acme" },
  agents: {
    analyst: defineAgent({
      instructions: "…",
      capabilities: ["analytics.query", "analytics.compile_metric"],
    }),
  },
  plugins: [analytics({ /* … */ })],
  // …
});
```

## Why one agent, not a supervisor + sub-agents

Production text-to-SQL systems have converged on the single-agent shape: **Dataherald** (single ReAct + tools), **Vanna 2.0+** (single agent with `run_sql` and `visualize_data` as **tools**, not sub-agents), **WrenAI** (typed workflow with deterministic steps; LLM at composition points only). None of them use a supervisor that re-verbalizes data to a coder sub-agent.

The reason is structural. When a supervisor delegates "pull the data, then write the report" to two sub-agents:

```
user prompt
  ↓
supervisor LLM
  ↓ delegates to sql-agent (tool call)
sql-agent LLM runs execute_postgres → rows live in sql-agent's scratchpad
  ↓ returns text (a prose paraphrase of the rows)
supervisor LLM (sees only the paraphrase, NOT the original rows)
  ↓ delegates to coder (tool call)
coder LLM (sees the supervisor's paraphrase, NOT the rows)
  ↓ writes file
```

Every arrow between LLMs is a **prose summary boundary**. The rows that came out of Postgres are now a narrative the supervisor wrote — and weak models fill gaps in that narrative with plausible-sounding numbers.

The single-agent shape eliminates the boundary:

```
user prompt
  ↓
agent LLM runs execute_postgres → rows in scratchpad
  ↓
agent LLM calls mastra_workspace_write_file with rows from scratchpad
  ↓
file on disk
```

Rows never leave the agent's working memory. There's no model that has to re-verbalize them.

## What this looks like in practice

A composite prompt — "pull yesterday's outlet KPIs and write me an HTML report" — runs as one agent turn with a sequence of tool calls visible in the trace:

```
── tool-call[0] execute_postgres
   input: { sql: "WITH yesterday AS (...) SELECT ..." }
── tool-result[0]
   output: { rows: [{ outlet: "luminere-bistro", revenue: 4521.18, ... }, ...] }
── tool-call[1] mastra_workspace_write_file
   input: { path: "reports/eod.html", content: "<table><tr>...</tr></table>" }
── tool-result[1]
   output: { bytesWritten: 2841 }
```

The agent saw the rows. The agent wrote the file. One LLM, two tools, zero handoff.

## What about workflows?

For composite work where you want **deterministic steps** (validate input → fetch → render → email) rather than letting the LLM choose the tool sequence, use Mastra Workflows. Multi-step + cron-scheduled workflows on Mastra are **planned** — see [ADR 0001](/adr/0001-adopt-mastra-durable-execution/). Current `schedules` are single-prompt cron config.

## What about genuinely heterogeneous specialists?

If you're building something that routes between distinct domains — say, an analytics agent **and** a customer-support agent **and** a code-review agent — then the supervisor + agents-as-tools pattern (Mastra's canonical example) is the right shape.

Arivie is a **domain-neutral agent framework**; analytics is the flagship plugin. For analytics workloads the work is text-to-SQL with optional file output — one LLM, multiple tools. For multi-domain routing, construct a supervisor in your app code using Mastra's `agents:` parameter.

## What you give up (vs. supervisor + sub-agents)

| You lose | Why it matters less than it sounds |
|---|---|
| Per-agent model selection (cheap supervisor + heavy coder) | Each `defineAgent` entry can specify its own `model`. |
| Memory isolation between sub-agents | Scratch noise in a single agent's context is normal and small. |
| Path-scoped sandbox isolation | The workspace root is path-scoped when wired. Same guarantee, fewer moving parts. |

## What you keep

Everything that matters for analytics agents:

- Read-only DB role enforcement
- Owner-identity boundary (`app.id`)
- Per-source `execute_<source>` tools with row-limit + timeout
- `compile_metric` semantic-layer compilation
- Skills (SOP playbooks) in `skills/*/SKILL.md`
- Mastra Memory for cross-turn state
- Workspace filesystem with path-guard (when wired)
