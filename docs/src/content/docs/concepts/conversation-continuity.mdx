---
title: Conversation continuity & memory
description: Thread-scoped memory, session IDs, and multi-turn context across surfaces.
---

Arivie persists conversation history through Mastra Memory. The runtime **session** maps to the Mastra **thread**; `resolveUser` supplies the **resource** (typically `userId`). The same session ID across requests gives the agent full multi-turn context.

## How it works

Mastra Memory stores messages keyed by thread ID and resource ID. When the agent runs, Mastra loads prior messages in the same thread and includes them as context.

## `sessions.create` with a session id

Pass `session.id` to continue a conversation:

```ts
const handle = await arivie.sessions.create({
  agent: "analyst",
  prompt: "What about last week?",
  user: { userId: "u1", permissions: ["analytics:read"], dbRole: "arivie_reader" },
  session: { id: "conv-123" },
});

for await (const event of handle.stream) {
  // …
}
```

- `session.id` → Mastra Memory `thread` (groups messages)
- `user.userId` → Mastra Memory `resource`

If `session` is omitted, a fresh thread is created per run — no continuity.

## One-shot with continuity

`arivie.prompt` accepts the same `session` field:

```ts
const text = await arivie.prompt({
  agent: "analyst",
  prompt: "What about last week?",
  user: { userId: "u1", permissions: ["analytics:read"], dbRole: "arivie_reader" },
  session: { id: "conv-123" },
});
```

## HTTP handler

The durable session API accepts session metadata on `POST /sessions`. The built-in `POST /api/chat` route (Mastra `@mastra/ai-sdk`) accepts thread IDs in the AI SDK message protocol for web chat continuity.

### Streaming with continuity

SSE streaming works the same way — pass the session/thread id and the agent streams the response while maintaining memory context:

```bash
curl -X POST http://localhost:3000/api/chat \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{"messages":[{"role":"user","content":"How much revenue yesterday?"}],"threadId":"mon-day-1"}'
```

## CLI chat with history

`arivie chat` (Ink TUI) lists past conversations and resumes them by session id — no web server required.

## Memory scopes

| Scope | Thread pattern | Use case |
|-------|---------------|----------|
| **Personal** | `session.id` or generated per user | Per-user conversations |
| **Instance-global** | `schedule-<id>` | Scheduled single-prompt runs |
| **Channel-scoped** | Derived from event metadata | Trigger-driven conversations |

Channel subscriptions can dynamically scope memory using `instanceId` and `resourceId` functions that read from the event payload — for example, isolating conversations per GitHub repository.

## `AsyncLocalStorage` context

The `execute_<source>` tools need the current user's DB role without explicit threading. Arivie uses `AsyncLocalStorage` to carry `UserContext` through tool calls:

```ts
import { runWithUserContext } from "@arivie/core";

await runWithUserContext(user, async () => {
  // Inside this frame, execute_<source> sees `user.dbRole`
  const text = await arivie.prompt({ agent: "analyst", prompt, user });
});
```

`arivie.prompt`, `sessions.create`, and `arivie.handler` set user context automatically.
