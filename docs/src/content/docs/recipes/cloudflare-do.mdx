## When to use this

Deploy on **Cloudflare Durable Objects** when you want one isolated Arivie instance per owner with sub-second cold starts and idle survival. The Worker routes requests to `arivie.handler` (including built-in `POST /api/chat`); model keys and `DATABASE_URL` come from Worker `env` bindings, not `process.env`.

## Code snippet

```ts
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const ownerId = new URL(request.url).searchParams.get("owner") ?? "default";
    const id = env.ARIVIE_DO.idFromName(ownerId);
    const stub = env.ARIVIE_DO.get(id);
    return stub.fetch(request);
  },
};

// Inside the Durable Object — one defineArivie app per stub
async function handle(request: Request, env: Env): Promise<Response> {
  const arivie = await getOrCreateArivie(env);
  return arivie.handler(request);
}
```

## Run it

```bash
wrangler dev
curl -X POST http://127.0.0.1:8787/api/chat \
  -H 'Content-Type: application/json' \
  -d '{"messages":[{"role":"user","content":"How many customers?"}]}'
```

Deploy recipe details are in RFC-002 §4.11. One Arivie app per owner per DO stub — same boundary model as Node deployments.
